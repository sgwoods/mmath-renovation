# Release Process

This document defines the build and release discipline for the AbTweak
renovation project.

The goal is not to imitate a modern binary packaging pipeline. It is to create
repeatable, numbered restoration checkpoints that:

- reflect the verified state of the repository at a point in time
- are easy to revert to later
- preserve research context, not just code state
- keep the public project page aligned with meaningful milestones

## Recommendation

The current project stage is best treated as:

- late pre-release
- research-grade beta
- one restored baseline with strong evidence across many domains, but not yet a
  final historically complete release

That is why the current checkpoint is versioned as:

- `0.10.0-beta.1`

This says:

- the core restored system is substantial and worth naming
- we are not yet at release-candidate confidence
- `hanoi-4` and the alternate framework are still open enough that `1.0.0`
  would be premature

## Versioning Scheme

Use semantic-style versioning with explicit pre-release labels.

### Pre-1.0

- `0.y.z-beta.n`
  - use for research-grade restoration checkpoints
  - expected now
- `0.y.z-rc.n`
  - use only once the main publication-facing baseline is nearly complete
  - `hanoi-4` should be either reproduced or convincingly explained
- `0.y.z`
  - optional, but only for a very stable internal milestone

### 1.0 and later

- `1.0.0`
  - first historically grounded restored release of the operator-style
    AbTweak/Tweak environment
- `1.x.y`
  - post-restoration maintenance, reproducibility improvements, or documented
    baseline bug fixes
- `2.0.0`
  - use only if a broader unified environment absorbs the alternate
    `reset-domain` framework or introduces a clearly larger restoration scope

## What Gets Versioned

Every named release checkpoint should include:

- a git commit
- an annotated git tag such as `v0.10.0-beta.1`
- the checked-in [VERSION](/Users/stevenwoods/mmath-renovation/VERSION) file
- a changelog entry
- a release snapshot under `releases/<version>/`

The release snapshot is the formal build artifact for this repository. It is a
source-and-results checkpoint, not a binary bundle.

The `manifest.json` inside a release snapshot records the source commit that
was captured when the snapshot was generated. If the snapshot files themselves
are checked in, the administrative commit that adds them may be one step later.

## Release Snapshot Contents

Each `releases/<version>/` directory should contain at least:

- `manifest.json`
- `status.json`
- `benchmark-status.md`
- `current-status.md`
- `release-summary.md`

These are generated by
[scripts/create-release-snapshot.sh](/Users/stevenwoods/mmath-renovation/scripts/create-release-snapshot.sh).

## Suggested Release Cadence

Cut a numbered pre-release only when one of these is true:

1. a meaningful new reproduction milestone lands
2. the public status story changes materially
3. a formal diagnostic checkpoint should be easy to revisit later

Do not cut a numbered release for every small commit.

## Current Release Rule

For now, use this default rule:

- cut a new `beta` when the benchmark-family status, publication crosswalk, or
  formal `hanoi-4` state changes materially

## Release Procedure

1. Confirm the repo is in a coherent state.
2. Update [VERSION](/Users/stevenwoods/mmath-renovation/VERSION).
3. Add a matching entry to [CHANGELOG.md](/Users/stevenwoods/mmath-renovation/CHANGELOG.md).
4. Refresh key docs:
   - [docs/current-status.md](/Users/stevenwoods/mmath-renovation/docs/current-status.md)
   - [docs/next-steps.md](/Users/stevenwoods/mmath-renovation/docs/next-steps.md)
   - any benchmark-specific formal checkpoint that materially changed
5. Generate a release snapshot:

```sh
sh /Users/stevenwoods/mmath-renovation/scripts/create-release-snapshot.sh
```

6. Commit the release prep.
7. Create and push an annotated git tag from the release-prep commit:

```sh
git -C /Users/stevenwoods/mmath-renovation tag -a v0.10.0-beta.1 -m "AbTweak restoration 0.10.0-beta.1"
git -C /Users/stevenwoods/mmath-renovation push origin v0.10.0-beta.1
```

## Revert and Recovery Rule

If a later change needs to be backed out conceptually, the project should
revert by returning to a named release tag rather than hunting for an arbitrary
commit hash.

That is the main reason to maintain numbered tags even before `1.0.0`.

## Pages Update Rule

The public Pages site should be updated only for numbered checkpoints, not for
every internal experiment.

Current Pages location:

- [mmath-renovation.html](/Users/stevenwoods/GitPages/public/mmath-renovation.html)

When a new numbered release is cut, update that page with:

- release version
- release date
- short milestone summary
- current benchmark-family status
- a link to the matching git tag or release commit

The homepage entry should also be updated if the visible public milestone
changed materially:

- [index.html](/Users/stevenwoods/GitPages/public/index.html)

Recommended public wording for the current release stage:

- `0.10.0-beta.1: publication-aligned lower Hanoi families, broad operator-style coverage, and a much sharper hanoi-4 extension diagnosis`

## Naming Rule For New Generic Strategies

Do not fold new generic search strategies into the historical baseline release
name.

If a future non-domain-specific strategy extension is added, it should be named
separately inside the release notes and docs, for example:

- `AbTweak-1993/unsat-aware-ranking`
- `AbTweak-1993/balanced-left-wedge`

That keeps historical restoration releases distinct from later research
extensions.
